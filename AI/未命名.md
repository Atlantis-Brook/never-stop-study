# 【AI基础设施】别再自己调教大模型生成JSON了 —— instructor库实战

大模型是个好东西，当吃透了语言理解能力之后能够输出质量非常优秀的结果。即便并不直接使用聊天功能，借用大模型强大的语言理解能力可以对大量文本输出和提出的需求快速做出准确的答案。

那么大模型能否用在自动化的流程呢？答案是可以的。例如通过简单的API调用我就可以令大模型输出一个JSON序列化的数据，进而推至下游服务进行处理。

例如给定下面的代码，我可以很轻松的获取我想要的JSON序列化数据：

```
from openai import OpenAI

# API提供商和API密钥需要自备哦！
endpoint = ""
api_key = ""

model_name = "gpt-4o"

client = OpenAI(
    base_url=endpoint,
    api_key=api_key
)

prompt = """你是一个情绪检测器，根据给定的文本检测出文本的情绪。
你的输出应当没有任何其他解释性文字，只有JSON形式的输出，示例如下：

{
    "emotion": "happy"
}


其中，"emotion"的值应当为"happy"、"sad"、"angry"、"fear"、"surprise"、"neutral"中的一个。
"""

response = client.chat.completions.create(
    model=model_name,
    messages=[
        {"role": "system", "content": prompt},
        {"role": "user", "content": "我今天很开心"}
    ],
)

print(response.choices[0].message.content)
```

调教Prompt已经成为现代AI人的一项必备技能了。在Prompt当中明确说明你的诉求和格式，然后在消息列表中写进输入即可。此时大模型的输出是`{"emotion": "happy"}`。很正常不过了。

不过这样做有一个问题：大模型可能会提供不符合要求的格式。就在这个代码正式敲定之前，我的Prompt里对JSON格式的标注用Markdown的代码块圈了起来。毕竟大模型也很喜欢使用Markdown：

````
prompt = """你是一个情绪检测器，根据给定的文本检测出文本的情绪。
你的输出应当没有任何其他解释性文字，只有JSON形式的输出，示例如下：

```json
{
    "emotion": "happy"
}
```


其中，"emotion"的值应当为"happy"、"sad"、"angry"、"fear"、"surprise"、"neutral"中的一个。
"""
````

此时大模型可能会错误的输出如下：

````
```json
{
    "emotion": "happy"
}
```
````

这样的输出放入下游任务当中会率先导致JSON解析错误，影响后续流程。看到这里你的第一反应可能是“我做个过滤不就可以了？”

情况不止于此。当上下文过长、Prompt错误配置、对要求输出的JSON格式复杂等诸多情况下，大模型的JSON输出可能并不理想。有可能会出现JSON的枚举值并不符合要求、突然冒出来说明文本破坏结构等不可预知的JSON输出问题。这也意味着这种情况下，大模型的JSON输出是不可信的。

不过好消息是：现在部分大模型提供商已经提供了JSON输出模式，如OpenAI：

```
prompt = """你是一个情绪检测器，根据给定的文本检测出文本的情绪。
你的输出应当没有任何其他解释性文字，只有JSON形式的输出，要求有emotion、reason、confidence三个字段。
其中，"emotion"的值应当为"happy"、"sad"、"angry"、"fear"、"surprise"、"neutral"中的一个。
"reason"的值应当为文本中情绪的来源，例如"因为今天天气很好"、"因为工作很顺利"、"因为和朋友一起玩"等。
"confidence"的值应当为0到100之间的整数，表示情绪检测的置信度。
"""

response = client.chat.completions.create(
    model=model_name,
    messages=[
        {"role": "system", "content": prompt},
        {"role": "user", "content": "广州朋友给请的干炒牛河真的太好吃了！"}
    ],
    response_format={
        "type": "json_object"
    }
)

print(response.choices[0].message.content)
```

此时大模型的输出为：

```
{
  "emotion": "happy",
  "reason": "因为享用了美味的干炒牛河",
  "confidence": 95
}
```

此时的输出可以保证是一个有效的JSON序列化数据，可以正常被各大解析器解析。但这又延伸出第二个问题：如果要求的JSON架构过于复杂，该怎么办？毕竟总不能每一次在Prompt当中手动输入每一个字段的含义和要求内容，并且当上下文过长时又会出现无法输出想要的JSON结构的问题。

到这里我们先把大模型放到一边，先思考一下我们在正常情况下（特别是后端校验），是如何验证用户输入的JSON是符合要求的？

例如我所期待的JSON架构为`{"confidence": 97}`，即confidence字段希望是一个数字时，用户输入了`{"confidence": "good"}`。那么这就是一个校验无法通过的JSON结构。只要是支持JSON作为数据格式的HTTP服务器，都会有JSON格式校验，但方式千秋，且和编程语言本身的特性强耦合。

有没有一种形式化的标准，能够对JSON的格式要求做出符号化？这就是JSON Schema。

JSON Schema通过JSON语言，对想要约束的JSON格式作出规定，它可能长这个样子：

```
{
    "name": "emotion_detection",
    "schema": {
        "type": "object",
        "properties": {
            "emotion": {
                "type": "string",
                "enum": [
                    "happy",
                    "sad",
                    "angry",
                    "fear",
                    "surprise",
                    "neutral"
                ]
            },
            "reason": {
                "type": "string",
                "description": "文本中情绪的来源，例如\"因为今天天气很好\"、\"因为工作很顺利\"、\"因为和朋友一起玩\"等。"
            },
            "confidence": {
                "type": "number",
                "description": "情绪检测的置信度，0到100之间的整数。",
                "minimum": 0,
                "maximum": 100
            }
        },
        "required": [
            "emotion",
            "reason",
            "confidence"
        ]
    }
}
```

一旦规定JSON的格式被语义化成了符号，那么这个格式规定作为输入本身也可以约束大模型。OpenAI旗下的模型支持使用JSON Schema，对输出进行约束。

```
prompt = """你是一个情绪检测器，根据给定的文本检测出文本的情绪。
你的输出应当没有任何其他解释性文字，只有JSON形式的输出，请按照JSON Schema的要求输出。
"""

response = client.chat.completions.create(
    model=model_name,
    messages=[
        {"role": "system", "content": prompt},
        {"role": "user", "content": "广州朋友给请的干炒牛河真的太好吃了！"}
    ],
    response_format={
        "type": "json_schema",
        "json_schema": {
            "name": "emotion_detection",
            "schema": {
                "type": "object",
                "properties": {
                    "emotion": {
                        "type": "string",
                        "enum": ["happy", "sad", "angry", "fear", "surprise", "neutral"]
                    },
                    "reason": {
                        "type": "string",
                        "description": "文本中情绪的来源，例如\"因为今天天气很好\"、\"因为工作很顺利\"、\"因为和朋友一起玩\"等。"
                    },
                    "confidence": {
                        "type": "number",
                        "description": "情绪检测的置信度，0到100之间的整数。",
                        "minimum": 0,
                        "maximum": 100
                    }
                },
                "required": ["emotion", "reason", "confidence"]
            }
        }
    }
)

print(response.choices[0].message.content)
```

这已经足够好了。通过JSON Schema对大模型的输出进行约束，能让大模型帮我们省去JSON输出的调教。

但这有一个问题：JSON Schema输出约束功能只有部分厂商才能使用。OpenAI和Anthropic支持 JSON Schema格式输出，而国产大部分大模型均不支持（或仅支持没有JSON Schema约束的JSON格式输出）。

有没有一个通用的工具，能够1. 简化JSON Schema带来的冗长语法以简洁的方式约束JSON架构 2. 将基于JSON Schema的架构约束应用到其他大模型上？

Python库Instructor给出了答案。Instructor通过Pydantic使用地道的Python语言定义架构并输出对应的JSON Schema，将得到的架构约束传入大模型，当大模型输出的结果不符合架构约束时，它会告诉大模型哪里有问题并要求修改不断重试。

Instructor已经针对OpenAI等常见的LLM厂商配置好了对应的Prompt，可以稍微看一下：

```

def handle_json_modes(
    response_model: type[Any] | None, new_kwargs: dict[str, Any], mode: Mode
) -> tuple[type[Any] | None, dict[str, Any]]:
    """
    Handle OpenAI JSON modes (JSON, MD_JSON, JSON_SCHEMA).

    Kwargs modifications:
    - When response_model is None: No modifications
    - When response_model is provided:
      - Mode.JSON_SCHEMA: Adds "response_format" with json_schema
      - Mode.JSON: Adds "response_format" with type="json_object", modifies system message
      - Mode.MD_JSON: Appends user message for markdown JSON response
    """
    if response_model is None:
        return None, new_kwargs

    message = dedent(
        f"""
        As a genius expert, your task is to understand the content and provide
        the parsed objects in json that match the following json_schema:\n

        {json.dumps(response_model.model_json_schema(), indent=2, ensure_ascii=False)}

        Make sure to return an instance of the JSON, not the schema itself
        """
    )

    if mode == Mode.JSON:
        new_kwargs["response_format"] = {"type": "json_object"}
    elif mode == Mode.JSON_SCHEMA:
        new_kwargs["response_format"] = {
            "type": "json_schema",
            "json_schema": {
                "name": response_model.__name__,
                "schema": response_model.model_json_schema(),
            },
        }
    elif mode == Mode.MD_JSON:
        new_kwargs["messages"].append(
            {
                "role": "user",
                "content": "Return the correct JSON response within a ```json codeblock. not the JSON_SCHEMA",
            },
        )
        new_kwargs["messages"] = merge_consecutive_messages(new_kwargs["messages"])

    if mode != Mode.JSON_SCHEMA:
        if new_kwargs["messages"][0]["role"] != "system":
            new_kwargs["messages"].insert(
                0,
                {
                    "role": "system",
                    "content": message,
                },
            )
        elif isinstance(new_kwargs["messages"][0]["content"], str):
            new_kwargs["messages"][0]["content"] += f"\n\n{message}"
        elif isinstance(new_kwargs["messages"][0]["content"], list):
            new_kwargs["messages"][0]["content"][0]["text"] += f"\n\n{message}"
        else:
            raise ValueError(
                "Invalid message format, must be a string or a list of messages"
            )

    return response_model, new_kwargs
```

可惜的是，这个Prompt以英文编写，不过后续如果有定制化需求，可以参照源代码来写一个中文翻译版本。

接下来我们可以做一个实战，将刚才情绪检测的JSON架构定义换一个方式表达：

```
from openai import OpenAI
from pydantic import BaseModel, Field
import instructor
from enum import Enum

# API提供商和API密钥需要自备哦！
endpoint = ""
api_key = ""

model_name = "gpt-4o"

openai_client = OpenAI(
    base_url=endpoint,
    api_key=api_key
)

client = instructor.from_openai(openai_client)

class EmptionType(Enum):
    HAPPY = "happy"
    SAD = "sad"
    ANGRY = "angry"
    FEAR = "fear"
    SURPRISE = "surprise"
    NEUTRAL = "neutral"


class EmotionDetection(BaseModel):
    emotion: EmptionType = Field(description="文本中情绪的类型")
    reason: str = Field(description="文本中情绪的来源，例如\"因为今天天气很好\"、\"因为工作很顺利\"、\"因为和朋友一起玩\"等。")
    confidence: int = Field(description="情绪检测的置信度，0到100之间的整数。", ge=0, le=100)

response = client.chat.completions.create(
    model=model_name,
    messages=[
        {"role": "user", "content": "广州朋友给请的干炒牛河真的太好吃了！"}
    ],
    response_model=EmotionDetection
)

print(response)
print(response.model_dump_json())
print(f"类型：{response.emotion.value}，原因：{response.reason}，置信度：{response.confidence}")
```

此时控制台的输出为：

```
emotion=<EmptionType.HAPPY: 'happy'> reason='广州朋友请客吃到美味的干炒牛河' confidence=97
{"emotion":"happy","reason":"广州朋友请客吃到美味的干炒牛河","confidence":97}
类型：happy，原因：广州朋友请客吃到美味的干炒牛河，置信度：97
```

第一行代表，Pydantic已经将验证通过的输出直接转换成原生Python对象。而我们可以看到在架构定义当中，emotion字段是来自Python的原生枚举语法糖，而这也被Pydantic识别到并成功实现自动转换。

何为“有Python味的代码”？在我认为，将复杂留给自己，将简单、自由和责任交给客户的代码便是。Pydantic成功做到了这一点。Python的流行HTTP服务器FastAPI就是通过Pydantic来实现快速JSON架构验证。

第二行代表，继承`pydantic.BaseModel`的Python类自带输出JSON属性，方便输出到外部程序当中。

第三行代表，Pydantic生成的原生Python对象是可信的。

如果挖掘好Pydantic和Instructor的潜力，那么通过LLM生成结构化的数据将会极大提高效率！
